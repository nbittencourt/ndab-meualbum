## Falha 1 — `tests/pacotinhos/abrir-pacotinhos.spec.ts:100`

**Descrição**: `deve limpar campo e manter foco após adição bem-sucedida`

**Erro**:
```
Error: Falha ao criar usuário: 409
  at criarUsuario (tests/support/helpers.ts:12)
  at usuarioAtivo (tests/support/helpers.ts:31)
```

**Causa raiz**: A função `criarUsuario` em `tests/support/helpers.ts` gera o email do usuário de teste usando apenas `Date.now()` como sufixo (`teste+${Date.now()}@exemplo.com`). Com 2 workers rodando em paralelo, dois testes distintos podem chamar `Date.now()` no mesmo milissegundo, gerando o mesmo endereço de email. O segundo registro retorna 409 (conflito) porque o usuário já existe.

**Arquivo afetado**: `tests/support/helpers.ts:4`

**Correção**: Adicionar um componente aleatório ao sufixo do email para garantir unicidade mesmo com múltiplos workers simultâneos.

```diff
- email: `teste+${Date.now()}@exemplo.com`,
+ email: `teste+${Date.now()}_${Math.random().toString(36).slice(2, 8)}@exemplo.com`,
```

## Falha 2 — `tests/perfil/perfil.spec.ts:185`

**Descrição**: `deve excluir conta e redirecionar para landing com mensagem (RN-P27)`

**Erro**:
```
Error: expect(locator).toBeVisible() failed
Locator: getByText(/conta foi excluída/i)
Expected: visible
Timeout: 3000ms
Error: element(s) not found
```

**Causa raiz**: O callback `onSuccess` em `client/src/pages/ProfilePage.tsx` chama `useAuthStore.getState().setUser(null)` imediatamente antes de `window.location.href = '/'`. A atualização do Zustand dispara um re-render do React de forma assíncrona (mas antes da navegação completa), o que faz o `ProtectedRoute` detectar `user = null` e executar uma navegação SPA para `/`. O componente `LandingPage` é montado por essa navegação SPA, lê e **remove** o item `contaExcluida` do `sessionStorage`, e exibe a mensagem brevemente. Em seguida, `window.location.href = '/'` força um reload completo da página — mas o `sessionStorage` já está vazio, então a mensagem não aparece novamente.

O servidor já chama `res.clearCookie('token')` ao excluir a conta, portanto o reload completo da página garante que `checkAuth()` encontre uma sessão inválida e zere o estado de auth sem necessidade de chamar `setUser(null)` manualmente.

**Arquivo afetado**: `client/src/pages/ProfilePage.tsx:127`

**Correção**: Remover a chamada `useAuthStore.getState().setUser(null)` do callback `onSuccess`. O reload completo via `window.location.href = '/'` é suficiente para limpar o estado de autenticação, já que o cookie JWT foi invalidado pelo servidor.

```diff
  onSuccess: () => {
    setShowExcluirModal(false);
    sessionStorage.setItem('contaExcluida', '1');
-   useAuthStore.getState().setUser(null);
    window.location.href = '/';
  },
```
