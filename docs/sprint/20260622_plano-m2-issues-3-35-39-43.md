## #3 — Descontinuar o reconhecimento por câmera (OCR)  ·  Médio

### Contexto

O issue original relatava a câmera não abrindo (box em branco) e, após captura, um "número
2" fantasma sem referência. O dono decidiu (comentário de 2026-06-22):

> "O modo de reconhecimento não é uma necessidade dos usuários e deve ser descontinuado.
> Remove as referências ao uso de OCR no sistema."

Portanto, a resolução **não** é consertar o `getUserMedia`/Tesseract, e sim **remover** todo
o recurso de câmera/OCR. Hierarquia de fontes: a decisão do dono (issue) sobrepõe os
wireframes que ainda mostram o modo "Fotografar".

### Diagnóstico

|---|---|---|
| `client/src/components/CameraModal.tsx` | arquivo inteiro | Modal de câmera + carga dinâmica de `tesseract.js` |
| `client/src/pages/FigurinhasPage.tsx` | 12 | `import { CameraModal }` |
| `client/src/pages/FigurinhasPage.tsx` | 432–433 | estado `mode`/`showCameraModal` |
| `client/src/pages/FigurinhasPage.tsx` | 449–455 | `addCameraMut` (origem `CAMERA`) |

| `client/src/pages/FigurinhasPage.tsx` | 699, 737–746 | condicional `mode === 'DIGITAR'` (manter só o form) e bloco `mode === 'FOTOGRAFAR'` |

### Plano de implementação

## #35 — Figurinha colada não reconhecida / pills por álbum  ·  Médio

### Contexto

Na funcionalidade Figurinhas, uma figurinha repetida **já colada** aparece com status "Pode
colar". O esperado: status **"Colada"** quando já presente em **todos** os álbuns do tipo; e,
quando colada em apenas alguns, exibir em **quais** álbuns ela já está colada — via **pill**
no card de cada álbum da seleção de destino. O botão "Colar" continua disponível (colar por
cima é válido — RN-CF08/CF09).

### Diagnóstico

| Arquivo | Linha(s) | Causa |
|---|---|---|
| `server/src/routes/colar-figurinhas.ts` | 21–31 | `coladas` só é populado quando há `albumId` na query; sem ele, `elegibilidade` cai sempre em `PODE_COLAR` |
| `client/src/pages/FigurinhasPage.tsx` | 174–177 | `SecaoRepetidas` chama `getEstoque(albumIdCtx || undefined)`; `albumIdCtx` (= `albumIdFromUrl`) costuma estar vazio → contexto "todos" |
| `client/src/pages/FigurinhasPage.tsx` | 61, 89 | `canPaste`/`StickerStatusBadge` usam a `elegibilidade` mentirosa |

### Plano de implementação

**1. Backend — status agregado + detalhamento por álbum (`GET /estoque`)**

Quando **não** houver `albumId`:
- Carregar os álbuns ativos do usuário (`Album.find({ usuarioId, arquivadoEm: null })`) e,
  por figurinha, considerar apenas os álbuns cujo `tipo_album_id` corresponde ao
  `secaoId → tipoAlbumId` da figurinha (figurinha só é colável em álbuns do tipo certo).
- Computar, por item de estoque:
  - `coladaEm: { albumId, nome }[]` — álbuns (compatíveis) onde já existe `FigurinhaColada`.
  - `elegibilidade`: `JA_COLADA` se colada em **todos** os álbuns compatíveis (e houver ≥1);
    `PODE_COLAR` caso contrário. (Status "parcial" é comunicado pelas pills, não por um novo
    enum — mantém retrocompatibilidade.)
- Quando **houver** `albumId`, manter o comportamento atual (status para aquele álbum),
  porém também popular `coladaEm` para consistência da UI.
- Evitar N+1: uma agregação de `FigurinhaColada` agrupada por `figurinhaId`/`albumId` para
  os álbuns do usuário, montada em `Map` antes do `.map` dos itens.

**2. Tipos compartilhados (`shared/src/types/sticker.ts`)**

Estender `EstoqueItem` com campo **aditivo** (retrocompatível):

```ts
export interface EstoqueItem extends EstoqueFigurinha {
  figurinha: Sticker;
  elegibilidade: ElegibilidadeStatus;
  coladaEm?: { albumId: string; nome: string }[]; // álbuns onde já está colada
}
```

**3. UI (`FigurinhasPage.tsx` — `RepetidaRow` + modal de seleção)**

- `StickerStatusBadge` passa a refletir o status agregado (mostra "Colada" quando
  `elegibilidade === 'JA_COLADA'`).
- No `radiogroup` de seleção de álbum, cada card que estiver em `coladaEm` recebe uma **pill
  "colada"** (rótulo textual + cor; nunca só cor — WCAG/RN-CF19). Reusar o estilo de pill já
  presente (`PilhaTag`).

### Testes

- **Vitest (integração backend):** `colar-figurinhas.int.test.ts` — cenários: figurinha
  colada em 1 de 2 álbuns compatíveis → `PODE_COLAR` + `coladaEm.length === 1`; colada em
  todos → `JA_COLADA`; álbum de tipo diferente não conta.
- **E2E:** colar uma repetida em todos os álbuns e verificar que o card passa a exibir
  "Colada"; abrir a seleção e verificar a pill no álbum correto.

### Impacto em specs

- `spec_colar_figurinhas.md` §5.2 / RN-CF07/CF08: detalhar o status agregado "todos os
  álbuns" e introduzir a regra das pills por álbum (nova RN — ex.: RN-CF30).

## #43 — "+ Repetidas" rápido no card colado  ·  Pequeno/Médio

### Contexto

Ao colar figurinhas, o usuário quer também enviar ao bolo de repetidas as cópias extras que

largura do card; o "+ Repetidas" ocupa os outros **3/4**.

### Diagnóstico

| Arquivo | Linha(s) | Elemento |

### Plano de implementação

**1. Backend — novo endpoint de incremento**

`POST /colar-figurinhas/estoque/adicionar` (espelho de `estoque/descartar`): recebe
`figurinhaId` (ou `numero`); faz upsert em `EstoqueFigurinha` para `(usuarioId, figurinhaId)`
incrementando `quantidade` em 1 (`findOneAndUpdate(..., { $inc: { quantidade: 1 } }, { upsert: true })`).
Validar que a figurinha existe no catálogo.

**2. Client API (`client/src/lib/api.ts`)**

Adicionar `colarFigurinhasApi.adicionarRepetida(figurinhaId)` no padrão das mutations

**3. UI (`StickerCardAL1`, ramo `isColada`)**

Substituir o container do `⋮` por um flex row com dois filhos:

Adicionar prop `onAddRepetida` à assinatura de `StickerCardAL1` e à cascata de `SecaoGrid` e

às repetidas") e atualização da `QuantidadeBadge`.

## #39 — Compartilhar lista de faltantes por link público  ·  Grande

### Contexto

Botão (na AL1 e/ou no popup de faltantes) que **gera um link único (UUID)**. Quem recebe o
link **não precisa de autenticação**: vê uma página, como a lista de figurinhas de impressão,
com a lista atual de faltantes. Se o usuário já gerou um link antes, o sistema **reutiliza** o
token persistido (sem gerar duplicatas).

### Diagnóstico / infraestrutura existente

- `Album` (`server/src/models/Album.ts`) não tem token de compartilhamento.
- Não há nenhuma rota pública de **dados** hoje (todas as rotas de domínio usam
  `requireAuth`); rotas públicas só existem no client (`App.tsx:104–110`).
- A lógica de faltantes já existe: `GET /:id/faltantes` (`albums.ts:179`) e o grid completo
  em `GET /:id/figurinhas` (`albums.ts:201`).
- O layout de impressão/grade está em `ListaFigurinhasModal.tsx` (componentes `PrintContainer`
  / `SecaoBlock`) — candidato a extração para reuso.

### Plano de implementação

**1. Modelo (`Album.ts`)**

Adicionar `shareToken: { type: String, default: null, index: { unique: true, sparse: true } }`.
Geração on-demand com `crypto.randomUUID()`.

**2. Backend**

- `POST /albums/:id/share` (autenticado): se `album.shareToken` é `null`, gera e persiste;
  retorna `{ token, url }`. Idempotente (reutiliza token existente — atende o requisito de
  não duplicar).
- `DELETE /albums/:id/share` (autenticado): zera `shareToken` (revogação — exigência LGPD).
- `GET /public/faltantes/:token` (**sem `requireAuth`**): resolve o álbum por `shareToken`,
  reusa a lógica de `/:id/figurinhas` e devolve **apenas** dados não-PII: tipo de álbum, nome
  personalizado, e a grade de figurinhas (números + status colada/faltante/repetida).
  Montar este router **fora** do guard de auth no app (`server/src/app.ts`/`index.ts`).

**3. Client**

- `App.tsx`: rota pública nova `"/faltam/:token"` (ao lado de `/politica-de-privacidade`),
  fora do `ProtectedRoute`.
- Extrair a grade/impressão de `ListaFigurinhasModal` para um componente compartilhado
  (ex.: `FigurinhasGrid`/`FigurinhasPrint`) consumido tanto pelo modal quanto pela página
  pública — evita duplicar o layout.
- Botão "Compartilhar faltantes" na AL1 (e/ou no cabeçalho do popup) que chama
  `POST /share`, exibe o link e oferece copiar.

**4. Pré-requisito LGPD (registrar antes de codar)**

- A página pública **não** expõe PII (sem nome/e-mail/`publicId` do dono; só tipo de álbum +
  números). Token opaco (UUID v4), **revogável** via `DELETE /share`.
- Registrar a decisão em `spec_albums.md` e referenciar `docs/legal/lgpd_guia_sistemas.md`
  (base legal: legítimo interesse / consentimento ao gerar o link). Sem essa decisão
  registrada, o item não entra em codificação (regra cross-cutting do `CLAUDE.md`).

### Testes

- **Vitest (integração):** `POST /share` idempotente; `GET /public/faltantes/:token` retorna
  faltantes corretos para token válido, 404 para token inexistente/revogado; ausência de PII
  no payload.
- **E2E:** gerar link na AL1, abrir a rota pública em contexto **sem sessão** (sem cookie de
  auth) e verificar a lista; revogar e confirmar 404.

### Impacto em specs

- `spec_albums.md`: nova seção "Compartilhamento de faltantes" + RNs (token persistido/
  reutilizado, acesso público sem auth, revogação, ausência de PII).
